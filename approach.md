### SHL Assessment Recommender — Approach Document

## Architecture & Agent Design
The system is a stateless FastAPI service. Every POST /chat call carries the full conversation history and produces an independent response. Internally, it uses a two-stage LLM pipeline:

1. Analyze: Extracts structured criteria (role, skills, seniority, test types) and classifies the turn into one of five actions: Clarify, Recommend, Refine, Compare, or Refuse.
2. Select: Given a list of retrieved catalog candidates, picks the final 1–10 shortlist with a brief justification for each.

The most critical design choice is Eager Recommendation. A single role mention (e.g., "Java developer") is enough to trigger a search, rather than asking follow-up questions. This conserves the strict 8-turn budget and handles users who volunteer information across multiple turns. If the initial shortlist needs adjustment, the Refine action handles it. When the agent does clarify, it asks exactly one focused question to avoid wasting turns.

## Retrieval Strategy
1. Embedding: Each assessment is represented as a single string combining its name, test type, description, keywords, and job levels. This is encoded using all-MiniLM-L6-v2 (384-dim, L2-normalized).
2. Indexing: Vectors are stored in a FAISS IndexFlatL2. Because vectors are unit-normalized, L2 distance approximates cosine similarity. The index is a single file loaded at startup—no external database needed.
3. Reranking: The top-25 FAISS results go through a hybrid rerank: 0.65 × embedding_similarity + 0.35 × keyword_overlap. This catches exact skill matches (e.g., "SAP", "React") that dense embeddings sometimes miss.

## Prompt Engineering
All prompts use Groq's native response_format: json_object (Llama 3.1 70B) to eliminate parsing failures.

1. Analysis Prompt: Extracts criteria and infers implied skills (e.g., "Sales manager" → sales, management). Explicitly instructs that a pasted Job Description is always sufficient context to recommend.
2. Selection Prompt: Receives the formatted candidate list. The LLM returns only integer indices, ensuring every URL in the final response is mapped from the local catalog—never generated by the LLM.
3. Compare Prompt: Injects raw catalog JSON for the requested assessments into the context window and strictly forbids using prior knowledge.

## Scope & Safety

1. Grounding: URLs are strictly sourced from catalog.json via index mapping.
2. Refusals: Off-topic requests, prompt injections, and general HR advice are caught at the analysis stage, triggering a polite rejection with end_of_conversation: true.
3. Schema Compliance: Enforced via Pydantic defaults (recommendations=[], end_of_conversation=false), ensuring the response never breaks the automated evaluator even if the LLM drops a field.

## Evaluation & Iterations
I iterated against the 10 provided conversation traces, focusing on Recall@10 and behavior probes (e.g., "does it clarify on turn 1 for vague inputs?").

# What didn't work:

1. Pure BM25/Keyword search: Failed completely on paraphrased queries ("software engineer" vs "developer").
2. LangChain: Added unnecessary latency and made debugging difficult. Raw httpx + FAISS is much simpler.
3. Multi-question clarifications: Wasted the 8-turn budget. One question per turn is optimal.
4. Large retrieval pools (top-50): The selection LLM lost good candidates in the middle of long lists. Top-25 with reranking was the sweet spot.
5. Pure embeddings: Adding the keyword-overlap reranking was the single biggest booster for Recall@10.

## Stack & AI Tools
1. LLM: Groq (Llama 3.1 70B, free tier) for fast, sub-second inference.
2. Embeddings/Vector Store: sentence-transformers + FAISS CPU.
3. API: FastAPI + Uvicorn.
4. Deployment: Render/Railway free tier (512MB RAM is sufficient).
5. AI Tools: Cursor IDE was used for boilerplate generation and docstrings. All core logic, prompt design, and architectural decisions were done manually.
